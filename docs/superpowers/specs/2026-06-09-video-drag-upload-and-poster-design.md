# 视频拖拽上传与封面展示设计

## 背景

文章编辑器已支持通过工具栏按钮上传视频，也已有外部文件拖拽事件，但拖拽逻辑目前只接收图片。视频上传成功后，后端会异步使用 FFmpeg 截取第 1 秒生成封面，前端也有 `poster` 属性和转码状态轮询能力。

当前封面链路存在两个缺口：

1. 编辑已有文章时，`editData.videos` 中的 `poster` 没有注册到编辑器视频元数据，导致重新打开文章时封面丢失。
2. `GET /video/:id` 返回数据库中的封面文件名，前端轮询却将其直接写入 `<video poster>`，可能用不可访问的相对文件名覆盖上传接口返回的完整 URL。

## 目标

- 富文本模式和 Markdown 分栏模式均支持拖入图片或视频上传。
- 拖拽进入编辑器时显示明确覆盖层：“释放以上传图片或视频”。
- 同时拖入图片和视频时，按文件原顺序依次上传并插入。
- 非图片、非视频文件被跳过并提示，不阻塞其余合法文件。
- 上传后和重新打开文章时均展示后端生成的视频封面。
- 复用现有上传、校验、转码、轮询和清理机制。

## 非目标

- 不增加手动选择或裁剪视频封面的功能。
- 不在浏览器端生成视频封面。
- 不改变每篇文章最多 2 个视频、单视频最大 20MB 的限制。
- 不新增数据库字段或迁移。
- 不调整 FFmpeg 截帧时间和转码流水线。

## 交互设计

### 拖拽反馈

外部文件进入编辑器可投放区域时，在编辑区域上方展示半透明覆盖层：

- 主文案：`释放以上传图片或视频`
- 辅助文案：`视频最大 20MB，每篇最多 2 个`
- 使用现有主色、虚线边框和上传图标。
- 覆盖层不拦截拖放事件。
- 文件离开编辑器或完成投放后立即隐藏。
- ProseMirror 内部节点重排不显示覆盖层。

### 文件处理

投放后读取 `DataTransfer.files`，保留浏览器提供的原始顺序：

- 图片调用现有 `uploadImage` 命令。
- 视频调用现有 `uploadVideo` 命令。
- 每个文件完成上传和插入后再处理下一个文件。
- 首个文件使用拖放坐标对应的编辑器位置。
- 后续文件使用上一个文件插入完成后的当前选区，保证媒体连续且顺序稳定。
- 不支持的文件统一跳过；若存在不支持文件，显示一次汇总提示。
- 若全部文件都不支持，不触发上传。

Markdown 分栏模式下仍以底层 Tiptap 文档作为插入目标，现有 Markdown 同步负责生成图片 HTML 或稳定视频 token，并刷新右侧预览。

## 前端设计

### 统一拖拽编排

在 `TiptapEditor.vue` 中将只处理图片的 `handleDrop` 改为媒体队列编排：

1. 过滤出 `image/*` 与 `video/*`。
2. 按原顺序逐个调用对应命令。
3. 等待当前上传 Promise 完成。
4. 更新下一项的插入选区。

图片继续使用 `imageUpload.getUploadPromise(file)`。视频上传扩展增加同等的 Promise 存储接口 `videoUpload.getUploadPromise(file)`，不改变 Tiptap 命令同步返回布尔值的约定。

视频 Promise 表示上传请求和节点插入阶段完成，不等待最长 60 秒的后台转码轮询。轮询继续独立运行，完成后静默刷新封面。

### 覆盖层

`TiptapEditor.vue` 在分栏容器和富文本容器内共用一个拖拽覆盖层组件结构。`isDragging` 仅由外部文件拖拽切换，覆盖层通过 `pointer-events: none` 保证原事件继续落到容器。

### 封面元数据

- `IArticleVideo` 增加 `poster?: string | null`。
- `syncArticleVideoRegistryFromProps` 使用文章接口返回的 `video.poster`，不再固定写入 `null`。
- 上传响应和轮询响应中的完整 `poster` URL 继续写入视频节点与运行时 registry。
- 无封面或仍在处理时，视频保持现有黑色背景与原生 controls，不额外制造前端占位图片。

## 后端设计

保留现有上传响应、FFmpeg/ffprobe 后台流程和数据库字段。

仅规范 `GET /video/:videoId` 的返回数据：

- `poster` 有值时转换为 `${baseURL}/article/video/${posterFilename}`。
- `poster` 无值时返回 `null`。
- 保持 `transcode_status`、时长、尺寸等字段不变。

这样上传接口、文章详情接口和状态轮询接口对 `poster` 的语义统一为可直接访问的 URL。

## 错误处理

- 类型、大小、magic number、数量限制继续由现有上传扩展校验并展示现有文案。
- 单个文件上传失败后，队列继续处理后续文件。
- 视频处理最终失败时继续使用现有行为：提示失败、删除节点并清理 pending ID。
- 不支持文件只汇总提示一次，避免多文件拖入产生连续 toast。
- 数据库中没有封面或状态仍在处理中时返回 `poster: null`，前端不报错。

## 测试

### 前端自动化

- 外部视频拖入时调用 `uploadVideo`，并携带拖放位置。
- 图片和视频混合拖入时严格按原顺序等待并调用。
- 不支持文件被跳过，合法文件仍上传，且只出现一次汇总提示。
- 外部文件拖入时显示覆盖层，内部节点拖拽不显示。
- 视频上传扩展暴露并清理上传 Promise。
- 已有文章视频 registry 使用 `poster`。
- Markdown 预览输出的视频带 `poster`。

### 后端自动化

- `GET /video/:id` 将封面文件名转换成完整公开 URL。
- 无封面时返回 `null`。
- 现有视频控制器、服务和文章 SQL 测试继续通过。

### 验证

- 前端运行相关 Vitest、完整测试、类型检查和构建。
- 后端运行完整 `node --test` 测试。
- 使用本地前后端和应用内浏览器验证：
  - 拖入单个视频。
  - 混合拖入图片和视频。
  - 覆盖层显示与消失。
  - 上传完成后视频显示封面。
  - 刷新或重新进入文章后封面仍显示。

如真实上传需要当前登录态，优先复用应用内浏览器已有会话；只有无法自动获得有效会话时才请求用户协助手测。
